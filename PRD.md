# 股票监控 Android APP 第一版 PRD + 技术方案

## 1. 产品目标
做一个面向高频盯盘用户的 Android 提醒工具，不替代交易软件，专注“发现异动并第一时间提醒”。用户可通过股票名称或代码添加自选，系统持续监控最新价格；当股票在短时间内出现大幅波动，或涨跌幅超过用户设定阈值时，APP 通过声音及时提示，减少用户反复打开行情软件的成本。第一版目标不是做全量行情平台，而是把“加自选-设阈值-后台盯盘-稳定提醒”这条链路跑通。

## 2. 核心用户故事
1. 作为上班盯盘用户，我想通过输入“宁德时代”或“300750”快速添加股票，避免查代码。
2. 作为波段交易用户，我想给单只股票设置涨幅、跌幅、短时波动阈值，便于按自己策略提醒。
3. 作为忙碌用户，我希望 APP 在后台持续监控，并在触发条件时直接发声音，不依赖我一直停留在前台。
4. 作为谨慎用户，我希望看到提醒原因，例如“5 分钟涨超 2%”或“现价跌破 18.50”，避免无效告警。

## 3. MVP 功能清单
- 股票搜索与添加：支持名称/代码搜索，加入自选。
- 自选列表：展示名称、代码、最新价、涨跌幅、监控状态。
- 阈值配置：支持单只股票设置涨幅阈值、跌幅阈值、短时波动阈值（如 5 分钟内 ±X%）。
- 后台监控：周期拉取行情并本地判断是否触发。
- 声音提醒：触发后播放提示音，并显示本地通知。
- 提醒去重：同一条件在冷却时间内不重复轰炸。
- 异常状态提示：数据源异常、网络中断、后台权限不足时给出明确提示。

## 4. 非功能要求
- 稳定性：行情轮询、告警判断、通知链路要可恢复；单点接口失败不应导致整体监控中断。
- 后台监控：Android 需以前台服务 + 系统通知常驻，保证后台活性；对被系统杀进程场景要能自动恢复或提示用户加白名单。
- 异常处理：网络超时、接口限流、无交易时段、返回脏数据时要降级，不误报、不崩溃。
- 功耗：默认轮询频率按场景分档，交易时段建议 5-15 秒，非交易时段自动降频或暂停；限制无意义唤醒。
- 数据频率：第一版不追求毫秒级，目标是交易时段内“秒级到十秒级提醒”，优先稳定和可持续运行。

## 5. 关键风险与取舍
- 风险一：免费行情源稳定性不足。取舍：第一版先接 1-2 个公开数据源并做容错，不先上高成本商业源。
- 风险二：Android 后台限制严格。取舍：接受“必须开启通知权限/电池白名单/前台服务”作为 MVP 前提，先保证提醒可靠。
- 风险三：告警过多影响体验。取舍：加入冷却时间、交易时段限制、阈值最小步长，优先减少噪音。
- 风险四：跨市场复杂度高。取舍：第一版先支持 A 股，港美股放后续。

## 6. 技术选型建议（优先 Android + Flutter）
- 客户端：Flutter，原因是 UI 开发效率高，后续可低成本扩到 iOS。
- Android 原生能力：通过 Flutter Plugin 接入前台服务、通知、音频播放、开机自启与电池优化引导。
- 状态管理：Riverpod 或 Bloc，推荐 Riverpod，结构轻、易测试。
- 本地存储：Hive/Isar 保存自选、阈值、告警记录；MVP 可优先 Hive。
- 行情层：封装独立 QuoteService，支持多数据源切换、超时与重试；客户端先做拉取型监控。
- 后台任务：交易时段内由 Android Foreground Service 驱动轮询；Flutter 负责展示与配置，核心监控逻辑尽量抽成可复用模块。
- 埋点与日志：本地日志 + Crash 收集，便于定位漏提醒问题。

## 7. 第一版开发里程碑
- 里程碑 1（1 周）：完成原型、数据结构、股票搜索/自选列表。
- 里程碑 2（1 周）：完成阈值配置、行情拉取、告警判断核心逻辑。
- 里程碑 3（1-2 周）：完成 Android 后台监控、声音提醒、通知、异常处理。
- 里程碑 4（1 周）：联调与实盘测试，重点验证后台存活、误报率、功耗与提醒时延。
- 里程碑 5（数天）：灰度发布，收集真实用户对阈值、频率、提示音的反馈。

**一句摘要：第一版聚焦 A 股自选监控与稳定声音提醒，先把后台可持续盯盘和低误报做扎实，再扩展市场与高级策略。**
